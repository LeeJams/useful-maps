INTERACTIVE SYSTEM MAP

서비스 워커는 페이지와 네트워크 사이의 ‘중간 관리자’입니다

페이지가 보내는 모든 요청은 먼저 서비스 워커를 거칩니다. 그 요청을 네트워크로 보낼지, 캐시에서 꺼낼지, 즉석에서 만들어낼지를 코드로 결정하죠. 아래에서 전략을 선택하고 요청을 보내면 실제 이동 경로와 응답 출처가 함께 바뀝니다. 원하는 순서대로 비교하고, 활용 사례에서 각 패턴이 실전에서 어디에 쓰이는지 이어서 살펴보세요.

  • 별도 스레드DOM 접근 불가, postMessage로 대화
  • 설치형탭을 닫아도 남아서 다음 방문 때 즉시 개입
  • HTTPS 전용보안 컨텍스트(또는 localhost)에서만 동작
용어 사전 열기
Scope
서비스 워커가 가로챌 수 있는 URL 범위. /sw.js를 / scope로 등록하면 사이트 전체 요청을 담당합니다.
Controller
지금 이 탭의 요청을 실제로 가로채고 있는 활성 서비스 워커. 등록만 된 상태와는 다릅니다.
App Shell
오프라인에서도 화면 뼈대를 띄울 수 있게 install 단계에서 미리 캐싱해 두는 HTML·CSS·JS 묶음.
Cache Storage
서비스 워커가 쓰는 요청/응답 전용 저장소. DevTools > Application > Cache Storage에서 볼 수 있습니다.
fetch 이벤트
페이지가 요청을 보낼 때마다 서비스 워커에서 발생하는 이벤트. event.respondWith()로 응답을 가로챕니다.
waiting
새 버전이 설치됐지만 이전 워커가 아직 탭을 잡고 있어 교체를 기다리는 상태. skipWaiting()으로 건너뛸 수 있습니다.
