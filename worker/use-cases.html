화면에서 본 흐름은 실전에서 이렇게 쓰입니다

메인 화면에서 확인한 install·fetch·캐시 전략·업데이트 흐름을 실제 제품 기능과 연결했습니다. 각 배지는 해당 사례의 출발점이 되는 브라우저 동작이나 API를 보여줍니다.

오프라인에서도 켜지는 앱 (PWA)

install + offline fallback

install 때 앱 셸을 미리 캐싱하고, 문서 탐색이 실패하면 offline.html을 돌려주는 이 랩의 구조가 그대로 PWA의 뼈대입니다. 여기에 manifest.json과 아이콘만 더하면 브라우저가 ‘홈 화면에 설치’를 제안하는, 지하철에서도 켜지는 앱이 됩니다.

installcache.addAll()manifest.jsonbeforeinstallprompt 
const APP_SHELL = ["/", "/app.css", "/app.js", "/offline.html"];

self.addEventListener("install", (event) => {
  event.waitUntil(
    caches.open(VERSION).then((cache) => cache.addAll(APP_SHELL))
  );
});
// + manifest.json과 아이콘 → "설치되는" 웹앱

실전 예: 오프라인에서도 읽히는 뉴스·문서 앱, 현장(공장·매장)에서 네트워크가 불안정한 사내 도구, 앱스토어 심사 없이 배포하는 설치형 웹앱.

재방문을 순간 로딩으로 만드는 캐시 라우팅

fetch + cache routing

Playground에서 전략을 하나씩 골라봤다면, 실전은 요청 종류별로 전략을 자동 배분하는 것입니다. 폰트·라이브러리는 Cache First, 이미지는 Stale While Revalidate, API는 Network First — 이 라우팅 하나로 재방문 로딩이 수 초에서 수백 ms로 줄어듭니다.

fetchrequest.destinationcaches.match() 
self.addEventListener("fetch", (event) => {
  const req = event.request;
  if (req.url.includes("/api/"))
    return event.respondWith(networkFirst(req));
  if (req.destination === "image")
    return event.respondWith(staleWhileRevalidate(req));
  if (req.destination === "style" || req.destination === "script")
    return event.respondWith(cacheFirst(req));
});

실무에서는 이 라우팅을 선언형으로 감싼 Workbox를 많이 씁니다. 직접 짜본 지금이 Workbox 문서가 제일 잘 읽히는 시점입니다.

탭이 닫혀 있어도 도착하는 푸시 알림

새 이벤트: push

서비스 워커는 페이지와 독립된 스레드라서, 탭을 다 닫아도 브라우저가 워커를 깨워 push 이벤트를 전달할 수 있습니다. Push API로 구독을 만들고 서버가 발송하면, 워커가 받아 시스템 알림으로 띄웁니다. 웹이 네이티브 앱의 영역으로 들어가는 지점입니다.

pushPushManager.subscribe()showNotification()notificationclick 
self.addEventListener("push", (event) => {
  const data = event.data?.json() ?? {};
  event.waitUntil(
    self.registration.showNotification(data.title, { body: data.body })
  );
});

self.addEventListener("notificationclick", (event) => {
  event.notification.close();
  event.waitUntil(clients.openWindow("/inbox"));
});

주의: iOS Safari는 16.4+에서, 홈 화면에 설치된 웹앱에서만 푸시가 동작합니다. 알림 권한은 반드시 사용자 행동(버튼 클릭) 뒤에 요청하세요.

오프라인에 쓴 글, 연결되면 자동 전송

새 이벤트: sync

캐시가 오프라인에서 읽기를 구한다면, Background Sync는 쓰기를 구합니다. 전송에 실패한 메시지를 IndexedDB 큐에 쌓고 sync를 등록해 두면, 연결이 돌아왔을 때 (탭이 닫혀 있어도) 브라우저가 워커를 깨워 재전송합니다.

syncregistration.sync.register()IndexedDB 
// 페이지: 실패한 전송을 큐에 넣고 sync 예약
await saveToQueue(message);
await registration.sync.register("send-queue");

// sw.js: 연결이 돌아오면 브라우저가 sync 이벤트를 발사
self.addEventListener("sync", (event) => {
  if (event.tag === "send-queue") event.waitUntil(flushQueue());
});

실전 예: 채팅 전송, 폼 제출, 좋아요 큐잉. Chromium 계열만 지원하므로 (Safari 미지원) 온라인 복귀 시 재시도하는 폴백을 함께 둡니다.

나가는 요청을 중간에서 가공하는 프록시

respondWith()

respondWith()에는 어떤 Response든 줄 수 있습니다. 오프라인 응답을 즉석 생성하듯, 요청을 다른 URL로 바꾸거나, 가짜 응답을 주거나, 응답을 변형할 수 있어요. 백엔드 없이 개발할 때의 API 목킹, 실험군만 새 API로 보내는 A/B 라우팅, 이미지 포맷 협상에 씁니다.

event.respondWith()new Request()Response.json() 
self.addEventListener("fetch", (event) => {
  const url = new URL(event.request.url);

  // 백엔드가 아직 없을 때: API 목킹
  if (url.pathname === "/api/user") {
    return event.respondWith(
      Response.json({ name: "테스트 계정", plan: "pro" })
    );
  }
});

이 랩의 오프라인 토글(x-sw-lab-offline 헤더를 보고 일부러 실패시키기)도 사실 이 패턴입니다. 테스트 환경에서 네트워크 상황을 재현하는 도구로도 좋습니다.

조용한 배포, 예고된 업데이트

waiting → activate

waiting 상태를 UX로 다듬으면 ‘새 버전이 있어요’ 토스트가 됩니다. 즉시 skipWaiting하면 빠르지만 열려 있는 탭과 코드가 어긋날 위험이 있고, 사용자에게 맡기면 안전하지만 버전이 오래 공존합니다. 자산 파일명 버저닝과 함께 설계하는 것이 정석입니다.

updatefoundregistration.waitingskipWaiting()controllerchange 
registration.addEventListener("updatefound", () => {
  const worker = registration.installing;
  worker.addEventListener("statechange", () => {
    if (worker.state === "installed" && registration.waiting) {
      showToast("새 버전이 있어요", () =>
        registration.waiting.postMessage("SKIP_WAITING")
      );
    }
  });
});

이 레포의 CSS/JS 파일명이 app.v6.css처럼 버전을 달고 있는 이유가 바로 이 문제입니다 — 옛 워커가 캐싱해 둔 이전 파일과 새 HTML이 섞이지 않게 하는 asset versioning.

어디까지 되고, 뭘 조심해야 하나

한계와 주의사항

설계 전에 알아두면 며칠을 아끼는 것들

  • 첫 방문에는 개입 못 함설치가 끝난 뒤의 요청부터 가로챕니다. 첫 방문 성능은 서버·CDN의 몫이고, 서비스 워커는 재방문의 무기입니다.
  • DOM을 못 만짐별도 스레드라 화면을 직접 갱신할 수 없습니다. 이 랩의 Event Log처럼 postMessage로 페이지에 부탁해야 합니다.
  • 언제든 종료·재기동됨브라우저가 유휴 워커를 수시로 재웁니다. 전역 변수에 상태를 두면 사라지므로, 상태는 IndexedDB나 Cache에 저장하세요.
  • 저장소는 지워질 수 있음디스크가 부족하면 브라우저가 origin 데이터를 정리할 수 있습니다. 중요하면 navigator.storage.persist()를 요청하세요.
  • iOS 제약푸시는 iOS 16.4+ 그리고 홈 화면에 설치된 웹앱에서만. Background Sync는 Safari 미지원 — 폴백 설계가 필수입니다.
  • 캐시하면 안 되는 것결제, 인증 토큰, 실시간 시세는 Network Only로. 개인 정보가 담긴 응답을 캐시에 넣을 때는 로그아웃 시 삭제까지 설계해야 합니다.
  • 캐시 무효화는 공짜가 아님버전이 붙은 캐시 이름 + activate에서 옛 캐시 삭제(이 랩의 코드) + 자산 파일명 버저닝, 세 가지가 세트입니다.
  • 등록 해제 ≠ 초기화unregister해도 Cache Storage는 남습니다. 이 랩의 Unregister 버튼이 caches.delete까지 같이 하는 이유입니다.
