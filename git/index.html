Rebase 되돌리기 그래프 보기

Git은 커밋 그래프를 만들고 이어 가는 도구입니다

브랜치, rebase, merge, revert는 모두 그래프의 모양을 바꾸는 선택입니다. 아래 그래프에서 같은 7개 커밋에서 나온 변경이 단계에 따라 어떻게 보이는지 비교해 보세요.

짧은 답

7개 커밋의 변경을 기능 단위로 되돌리기 쉽게 하려면 merge commit이나 squash commit을 남기세요.

  1. git merge --no-ff feature는 feature 병합 전체의 경계가 되는 merge commit을 남깁니다.
  2. 이 예시의 git revert -m 1 <merge_sha>는 1번 부모 D2를 mainline으로 삼아 M1과 D2 사이의 차이를 반대로 적용합니다.
  3. squash merge라면 git revert <squash_sha>처럼 일반 커밋 하나를 되돌리면 됩니다.
  4. merge revert 뒤 기능을 다시 넣으려면 revert 커밋을 다시 revert하거나, 현재 main에서 원래 변경 전체를 새 커밋으로 만드세요.

<merge_sha>처럼 꺾쇠로 표시한 값은 그대로 실행하지 말고 실제 SHA로 바꾸세요.

그래프 실험실 기본 개념 Rebase Merge/Revert 명령어

시나리오: feature 브랜치에 7개 커밋이 있습니다

결과를 선택하면 그래프와 오른쪽 해설이 함께 바뀝니다. 커밋 노드를 누르면 각 노드의 의미도 확인할 수 있습니다.

커밋 노드를 누르면 설명이 바뀝니다.

main feature rebase 후 새 커밋 merge commit squash commit revert commit

기본 개념은 네 가지입니다

              
Git이 어렵게 느껴질 때는 명령어보다 그래프에서 무엇이 움직이는지 먼저 보세요.

            

            

              

                
                  
                
                
Commit

                
프로젝트의 스냅샷입니다. 각 커밋은 부모 커밋을 가리키기 때문에 선으로 이어집니다.

              

              

                
                  
                
                
Branch

                
커밋을 가리키는 이름표입니다. 새 커밋을 만들면 현재 브랜치 이름표가 앞으로 이동합니다.

              

              

                
                  
                
                
Merge

                
두 브랜치의 변경을 합칩니다. --no-ff를 쓰면 합친 순간을 나타내는 커밋이 남습니다.

              

              

                
                  
                
                
Rebase

                
내 변경을 최신 main 뒤에 새 커밋으로 다시 적용합니다. 부모가 달라지므로 SHA도 달라집니다.

              

            

          


          

            

              
Rebase는 내 커밋의 변경을 최신 main 위에 다시 적용하는 작업입니다

              
그래프는 깔끔해지지만 커밋 ID가 바뀝니다. 그래서 혼자 쓰는 feature 브랜치에서 가장 안전합니다.

언제 쓰면 좋나요

                  
feature 브랜치를 혼자 쓰고 있고, 최신 main 위에서 테스트하고 싶을 때 좋습니다.

                

                

                  
언제 피해야 하나요

                  
이미 팀원이 가져간 공유 브랜치에서는 조심하세요. rebase는 기존 커밋을 새 커밋으로 다시 만들기 때문입니다.

                

                

                  
merge 전 팁

                  
rebase 후 fast-forward로 합치면 main에는 새로 만들어진 7개 커밋이 일렬로 들어갑니다. 기능 전체를 되돌릴 경계가 필요하면 --no-ff나 squash를 선택하세요.

                

                

                  
이미 push했다면

                  
혼자 쓰는 원격 feature를 rebase한 경우에도 팀 정책을 먼저 확인하세요. 원격 갱신에는 일반 --force 대신 git push --force-with-lease가 필요할 수 있습니다.

                

              

            

          


          

            

              
7개 커밋을 되돌리기 쉽게 합치는 방법

              
테스트 실패에 대비하려면 “main에 어떤 커밋을 남길지”가 중요합니다.

            

            

              

                
                  

                    전략
                    main에 남는 모양
                    테스트 실패 시
                    추천 상황
                  

                
                
                  

                    Merge commit
되돌리기 기준 명확		
                    F1~F7 기록은 보존되고, main에는 M1 merge commit이 생깁니다.
                    git revert -m 1 <merge_sha>에서 1번 부모를 mainline으로 삼아 M1과 그 부모 사이의 차이를 반대로 적용합니다. merge 기록과 ancestry는 남습니다.
                    커밋 기록과 기능 병합의 경계를 함께 남기고 싶을 때. revert 뒤 기존 feature에 fix만 추가해 다시 merge하면 원래 변경은 돌아오지 않습니다. 먼저 revert commit을 revert하거나 현재 main에서 전체 변경을 새 커밋으로 만드세요.
                  

                  

                    Squash merge
main이 간결함		
                    F1~F7 변경이 S1 커밋 하나로 합쳐져 main에 들어갑니다. merge 관계는 남지 않습니다.
                    git revert <squash_sha>로 일반 커밋 하나를 되돌립니다.
                    main을 기능 단위로 읽고 싶고, 중간 커밋이 main에 남지 않아도 될 때. 같은 브랜치를 나중에 일반 merge하면 이미 합쳐진 것으로 자동 인식되지 않을 수 있습니다.
                  

                  

                    Rebase + ff-only
범위 지정 필요		
                    main에 F1'~F7'이 일렬로 들어갑니다.
                    연속된 선형 구간이라면 git revert --no-commit <first_sha>^..<last_sha>로 first부터 last까지의 역패치를 모은 뒤 검토하고 한 번 커밋합니다.
                    작업 트리와 index가 깨끗하고, 되돌릴 연속 범위를 정확히 확인할 수 있을 때.
                  

                  

                    Rebase 없는 ff-only
현재 시나리오에서는 실패		
                    main과 feature가 이미 갈라졌으므로 git merge --ff-only feature/login은 병합을 거부합니다.
                    먼저 feature를 최신 main 위에 rebase한 뒤 ff-only로 반영해야 일렬의 그래프가 됩니다.
                    main이 feature의 조상이고 두 브랜치가 갈라지지 않았을 때만 바로 가능합니다.
                  

                
              

            

          


          

            

              
실전 명령어

              
먼저 git status와 범위를 확인하세요. 이 페이지의 main은 저장소에 따라 master일 수 있습니다.

            

            

              

                
최신 main 위에서 다시 테스트

                
혼자 쓰는 feature 브랜치일 때 사용하세요.

                
git switch feature/login
git fetch origin
git rebase origin/main

# 충돌 해결 후
git status
git add path/to/resolved-file
git rebase --continue

              

              

                
merge commit을 남겨 합치기

                
기능 병합의 경계를 남기고 싶을 때.

                
git status --short
git switch main
git pull --ff-only
git merge --no-ff feature/login -m "merge: feature/login"
MERGE_SHA=$(git rev-parse HEAD)

# 테스트 실패 시 부모 순서 확인 후
git show --pretty=raw "$MERGE_SHA"
git revert -m 1 "$MERGE_SHA"

              

              

                
연속된 커밋을 한 revert로 묶기

                
abc1234와 def5678을 실제 첫 SHA와 마지막 SHA로 바꾸세요.

                
FIRST_SHA=abc1234
LAST_SHA=def5678

git status --short
git log --reverse --oneline "${FIRST_SHA}^..${LAST_SHA}"
git revert --no-commit "${FIRST_SHA}^..${LAST_SHA}"
git diff --cached
git commit -m "revert: feature/login"

공유 브랜치 rebase는 조심하세요

                  
이미 push했고 다른 사람이 가져갔다면 rebase보다 merge가 보통 안전합니다.

                

              

              

                
                  
                
                

                  
공유된 main에서는 보통 revert가 낫습니다

                  
reset 후 force-push하는 것보다 협업 기록을 보존하기 쉽습니다. revert도 충돌할 수 있으므로 복구 브랜치에서 테스트한 뒤 PR로 반영하세요.
