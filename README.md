# 쓸모지도 (Useful Maps)

복잡한 개발 개념과 실전 팁을 **왜 필요한지 → 어떻게 작동하는지 → 직접 비교하기 → 실전에 적용하기** 순서로 보여 주는 인터랙티브 시각 가이드 라이브러리입니다.

개념을 글로 설명하는 데서 끝나지 않고, 흐름과 관계를 눈으로 보고 직접 바꿔 보면서 이해할 수 있게 만드는 것이 이 프로젝트의 목적입니다.

## 무엇을 만드는 곳인가요?

쓸모지도의 각 가이드는 하나의 기술적 질문을 다룹니다. 설명, 도식, 인터랙션, 실전 예시가 따로 놀지 않고 하나의 학습 흐름으로 이어져야 합니다.

| 순서 | 질문 | 가이드가 제공해야 하는 것 |
| --- | --- | --- |
| **WHY** | 왜 알아야 하나요? | 이 개념이 해결하는 문제와 판단 기준 |
| **HOW** | 어떻게 작동하나요? | 상태, 데이터, 요청이 움직이는 순서와 관계 |
| **TRY** | 무엇이 달라지나요? | 같은 조건에서 선택지를 직접 바꾸고 비교하는 경험 |
| **APPLY** | 어디에 쓰나요? | 실제 명령어, 구현 패턴, 적용 상황과 주의점 |

시각 자료는 장식이 아니라 설명의 일부입니다. 선은 관계를, 노드는 상태를, 움직임은 처리 순서를 보여 주어야 합니다.

## 현재 가이드

| 가이드 | 직접 확인할 수 있는 것 |
| --- | --- |
| Git 그래프와 안전한 되돌리기 (`/git/`) | 같은 커밋이 rebase, 기본 merge, `--no-ff`, squash, fast-forward, revert에 따라 어떻게 달라지는지 비교합니다. |
| Service Worker 요청 경로 (`/service-worker/`) | 실제 Service Worker와 Cache Storage를 사용해 Network First, Cache First, Stale While Revalidate, Network Only를 실험합니다. |
| Service Worker 활용 사례 (`/service-worker/use-cases/`) | 오프라인 셸, 캐시 전략, 업데이트, 백그라운드 작업 등 실전 활용을 살펴봅니다. |
| BFS (`/algorithms/bfs/`) | 시작점을 바꾸고 큐, 방문 순서, 거리표를 한 단계씩 움직이며 무가중 그래프의 최단 간선 수를 확인합니다. |
| DFS (`/algorithms/dfs/`) | 재귀 호출과 명시적 스택, 이웃 순서를 바꾸며 깊이 들어가고 되돌아오는 흐름을 비교합니다. |
| 백트래킹 (`/algorithms/backtracking/`) | 4×4 N-Queens에서 선택, 충돌 검사, 가지치기, 되돌림, 첫 번째 해를 실제 탐색 순서로 따라갑니다. |

## 누구나 기여할 수 있습니다

새로운 가이드와 기존 가이드 개선을 Pull Request로 받습니다.

다음과 같은 기여를 환영합니다.

- 어렵거나 자주 헷갈리는 개발 개념을 새로운 시각 가이드로 추가
- 기존 설명의 정확성, 흐름, 예제, 명령어 개선
- 비교하기 어려운 선택지를 더 명확한 도식이나 인터랙션으로 개선
- 접근성, 키보드 조작, reduced motion, 모바일 사용성 개선
- 재현 가능한 버그 수정

큰 가이드는 구현 전에 Issue나 PR 설명으로 주제와 핵심 질문을 먼저 공유해 주세요. 이미 있는 가이드와 겹치는지, 하나의 화면에서 무엇을 비교할지 먼저 맞추면 리뷰가 훨씬 빨라집니다.

## 좋은 가이드의 기준

### 하나의 질문에 집중합니다

한 페이지가 너무 많은 개념을 얕게 훑기보다, 사용자가 실제로 결정해야 하는 한 가지 질문을 깊이 보여 줍니다.

### 이유를 먼저 설명합니다

용어와 API 목록보다 “이것이 없으면 어떤 문제가 생기는가”를 먼저 보여 줍니다. 사용자는 필요성을 이해한 뒤에 작동 방식을 배워야 합니다.

### 시각 자료가 관계를 설명합니다

선, 노드, 단계, 표, 타임라인은 상태와 인과관계를 드러내야 합니다. 설명을 가려도 의미가 없는 그림이라면 장식에 가깝습니다.

### 직접 비교할 수 있어야 합니다

선택지를 바꿨을 때 그래프, 경로, 결과, 설명이 함께 바뀌어야 합니다. 비교 대상은 가능한 한 같은 초기 조건을 사용합니다.

### 실전으로 이어집니다

마지막에는 실제 명령어, 코드, 사용 상황, 실패 시 복구 방법처럼 바로 적용할 수 있는 정보를 제공합니다.

### 누구나 같은 정보를 얻을 수 있어야 합니다

키보드, 작은 화면, 화면 읽기 도구, reduced motion 환경에서도 핵심 내용과 결과가 사라지면 안 됩니다.

## 새 가이드 추가하기

1. `src/content/guides.ts`의 slug 타입과 `guides` 목록에 메타데이터를 추가합니다.
2. `src/app/<slug>/page.tsx`를 만들고 공통 `GuideHero`를 사용합니다.
3. 페이지를 WHY → HOW → TRY → APPLY 순서로 구성합니다.
4. 상태나 브라우저 API가 필요한 실험만 별도 Client Component로 분리합니다.
5. 홈 목록, 헤더 라벨, 페이지 metadata, 가이드 안의 상태 수와 설명을 동기화합니다.
6. 데스크톱, 모바일, 키보드, 반복 실행을 검증합니다.

주요 구조는 다음과 같습니다.

```text
src/
├── app/                         # 페이지와 가이드 라우트
│   └── algorithms/              # BFS, DFS, 백트래킹 알고리즘 가이드
├── components/                  # 공통 헤더, 로고, GuideHero, 아이콘
└── content/guides.ts            # 홈과 내비게이션의 가이드 목록

public/service-worker/           # Service Worker 실험에 필요한 공개 파일
```

세부 구현 규칙과 Service Worker 안전 규칙은 [`AGENTS.md`](./AGENTS.md)를 확인해 주세요.

## 로컬에서 실행하기

```bash
npm install
npm run dev
```

브라우저에서 `http://localhost:3000`을 엽니다. Service Worker 실험은 `localhost`에서 확인해 주세요.

변경을 제출하기 전에 다음 명령을 실행합니다.

```bash
npm run lint
npm run build
npm audit --omit=dev
```

## Pull Request 체크리스트

- [ ] 한 PR이 하나의 가이드 또는 하나의 명확한 개선에 집중합니다.
- [ ] WHY → HOW → TRY → APPLY 흐름이 드러납니다.
- [ ] 기술 설명과 명령어를 신뢰할 수 있는 자료로 확인했습니다.
- [ ] 시각 자료가 장식이 아니라 상태, 관계, 순서를 설명합니다.
- [ ] 선택 상태와 설명, 범례, 결과가 서로 일치합니다.
- [ ] 인터랙션을 첫 실행과 반복 실행 모두 확인했습니다.
- [ ] 키보드만으로 주요 기능을 사용할 수 있습니다.
- [ ] reduced motion에서도 같은 정보를 얻을 수 있습니다.
- [ ] 1440px 데스크톱과 390px 모바일에서 확인했습니다.
- [ ] 모바일에서 문서 전체의 가로 overflow가 없습니다.
- [ ] 정상 흐름에서 브라우저 console error가 없습니다.
- [ ] `npm run lint`, `npm run build`, `npm audit --omit=dev`를 통과했습니다.
- [ ] 시각 변경이 있다면 리뷰어가 비교할 수 있는 스크린샷이나 짧은 영상을 첨부했습니다.

작고 명확한 개선도 좋은 기여입니다. 더 많은 사람이 어려운 개념을 “읽어서”가 아니라 “보면서” 이해할 수 있도록 함께 지도를 넓혀 주세요.
